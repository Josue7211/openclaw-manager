## Certification Contract

This matrix is the source of truth for the ClawControl Portainer module. A row is complete only when it has implementation evidence, automated test evidence, and live certification evidence against a disposable Portainer-backed environment, unless the row is explicitly marked `be-only` or `blocked-upstream`.

Allowed statuses:

- `pass`: implemented and certified against a live Portainer CE environment.
- `fail`: missing, broken, untested, or lacking evidence.
- `be-only`: Business Edition-only behavior, shown as unavailable/locked in ClawControl.
- `blocked-upstream`: unavailable because the connected Portainer instance, endpoint type, credentials, or lab environment does not expose it.

No broad area may be claimed complete while any CE row is `fail`.

## Current Repo Baseline

Existing code already provides useful building blocks:

- Portainer instance config parsing and secret storage in `src-tauri/src/routes/homelab.rs`.
- Dedicated Portainer gateway routes: `/api/homelab/portainer/instances`, `/api/homelab/portainer/environments`, `/api/homelab/portainer/capabilities`, `/api/homelab/portainer/resources`, and `/api/homelab/portainer/action`.
- Docker/Portainer inventory for endpoints, stacks, containers, images, volumes, networks, secrets, configs, and registries.
- Runtime snapshots for version/edition probes, system status, settings availability, endpoint platform, Swarm detection, groups, and tags; native operations can now create/update/remove endpoint groups and tags through server-side gateway actions.
- Admin inventory now includes users, teams, app templates, and custom templates in backend resources/capabilities and in the dedicated Admin UI; native operations can create/update/remove Portainer users, teams, and custom templates through server-side gateway actions, plus retrieve app/custom template files.
- Read-only Swarm inventory now includes services, nodes, and tasks for Swarm-active Docker endpoints.
- Swarm service/node/task inspect and log actions, service scale/remove, and node availability updates are routed through `/api/homelab/portainer/action` without exposing Portainer tokens to the browser.
- Basic Portainer actions through `/api/homelab/control`.
- Typed destructive confirmations and audit logging for homelab controls.
- xterm.js and CodeMirror dependencies are already available in the frontend.
- Dedicated Portainer stack updates use CodeMirror-backed compose and environment editors before calling `/api/homelab/portainer/action`.
- Dedicated Portainer container log sessions use `/api/homelab/portainer/terminal/session` plus `/api/homelab/portainer/terminal/ws` so xterm streams logs without exposing Portainer tokens to the browser.
- Portainer container exec can also start a server-side terminal session and bridge xterm stdin/stdout/stderr through Portainer's exec websocket without exposing Portainer credentials to the browser.
- Portainer Kubernetes pod exec creates a server-side `/api/homelab/portainer/terminal/session`, connects to Portainer's Kubernetes websocket proxy with `v4.channel.k8s.io`, maps stdin/stdout/stderr channels, and renders the shell in xterm without exposing Portainer tokens to the browser.
- Endpoint Docker events can be fetched as bounded/filterable history through the server-side Portainer action gateway, rendered in a structured event drawer with local filtering and row detail drill-in, and followed live through the server-side Portainer terminal websocket bridge. Live certification remains incomplete.
- Portainer Docker events certification is scripted by `npm run portainer:certify-live -- --read-only` for every matching configured Docker endpoint's normalized dashboard inventory shape, Docker secret/config shape, Swarm service/node/task shape when Swarm is present, bounded history, and websocket follow session proof. `npm run portainer:certify-live -- --create-disposable --yes` creates a guarded `clawcontrol-cert*` container, observes its event in the live stream and bounded history, then removes it unless `--keep-disposable` is set. Adding `--container-details` certifies inspect/logs/stats/processes/exec/filesystem-changes against the same disposable container. Adding `--container-mutations` certifies restart-policy update, resource update, rename, duplicate, recreate, and typed cleanup while tracking the recreated container id. Adding `--docker-images` certifies image pull/inspect/history/tag/tagged-inspect and removes only the temporary certification tag. Adding `--docker-assets` certifies guarded disposable volume create/inspect/remove and network create/inspect/connect/disconnect/remove against the same disposable container. Adding `--docker-stacks` certifies guarded disposable standalone stack create/inspect/file/logs/update/stop/start/delete. Adding `--container-lifecycle` certifies start/restart/pause/unpause/stop/start/kill with inspect-state checks. The script has mock coverage through `npm run portainer:test-certify-readonly` and `npm run portainer:test-certify-disposable`; the matrix rows remain `fail` until real disposable Portainer-backed runs are recorded.

These are not sufficient for completion. They are starting evidence only.
