       Add a useGatewaySSE call inside SessionsPage that:
       a. Subscribes to `['chat']` events
       b. Invalidates `queryKeys.gatewayActivity` for the activity feed
       c. Uses `onEvent` callback to trigger notifications for session lifecycle events

       ```typescript
       const handleSessionEvent = useCallback((eventName: string, payload: unknown) => {
         if (eventName !== 'chat') return
         const data = payload as Record<string, unknown> | null
         if (!data) return

         // Detect session completion or error from the event payload.
         // Gateway chat events include status/action fields -- handle defensively.
         const status = (data.status as string) ?? (data.action as string) ?? ''
         const sessionTask = (data.task as string) ?? (data.label as string) ?? 'Session'
         const agentName = (data.agentName as string) ?? (data.agent as string) ?? ''

         if (status === 'completed' || status === 'done') {
           addNotification(
             'system',
             'Session Completed',
             agentName ? `${agentName}: ${sessionTask}` : sessionTask,
             '/sessions',
           )
         } else if (status === 'error' || status === 'failed') {
           addNotification(
             'alert',
             'Session Error',
             agentName ? `${agentName}: ${sessionTask}` : sessionTask,
             '/sessions',
           )
         }
       }, [])

       useGatewaySSE({
         events: ['chat'],
         onEvent: handleSessionEvent,
         queryKeys: {
           chat: queryKeys.gatewayActivity,
         },
       })
       ```

       NOTE: The notification uses `addNotification` from the existing NotificationCenter
       store. The user's DND and notification toggle settings are respected by the
       NotificationCenter component itself -- callers just add to the store, the rendering
       layer checks preferences. No need to check preferences here.

       NOTE: The exact shape of gateway `chat` event payloads is not documented in the
       protocol spec, so we read `status`, `action`, `task`, `label`, `agentName`, and
       `agent` defensively with nullish fallbacks. This ensures the code works with
       whatever shape the gateway sends and won't crash on unexpected payloads.
