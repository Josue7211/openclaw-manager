**Atomic setWallbashState API eliminating race conditions from fragmented state updates, recv_timeout coalescing debounce replacing sleep-based event dropping, and gsettings monitor restart loop with exponential backoff**

## Accomplishments
- Added setWallbashState() that atomically updates colors + colorScheme + gtkThemeId with a single generation increment, preventing resolveThemeDefinition from reading partial state mid-update
- Reduced wallbashUpdatedRecently suppression window from 5000ms to 500ms so gsettings events arriving 1-2s after wallbash writes are no longer blocked
- Replaced wallbash watcher's sleep(150ms)+drain debounce with recv_timeout coalescing that fires 200ms after the LAST event in a burst
- Added restart loop with exponential backoff (1s-60s) to gsettings monitor subprocess with proper child process reaping

- `src-tauri/src/commands.rs` - recv_timeout coalescing debounce in wallbash watcher, restart loop in gsettings monitor

## Decisions Made
- 500ms suppression window chosen because 5000ms was suppressing legitimate gsettings events that arrive 1-2s after wallbash file writes; 500ms is enough to coalesce the file watcher's own burst
- recv_timeout coalescing replaces sleep+drain pattern because the old approach dropped events arriving within 250ms of the last emit, while recv_timeout properly coalesces by extending the window
- Exponential backoff 1s to 60s with reset after 30s stable run balances fast recovery against thrashing
- Individual setters (setWallbashColors, setWallbashColorScheme) kept for backward compatibility since main.tsx startup code still calls them independently
