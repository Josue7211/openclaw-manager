## Phase Boundary

Fix the broken wallbash dark↔light theme switching that was introduced via 7 incremental patches during phase 03 verification. The patches individually address parts of the problem but don't work together reliably. This phase audits all 7 commits, identifies root causes, and rebuilds the wallbash integration properly.

</domain>

<decisions>
## Implementation Decisions

### Approach
- Audit all 7 wallbash-related commits from phase 03 and determine what to keep, fix, or revert
- The gsettings monitor subprocess approach is sound but may need hardening (error handling, reconnection)
- The debounced apply pattern is correct but 100ms may be too aggressive — tune based on testing
- useGtkTheme toggle and auto-detection logic needs localStorage state machine audit
- Crossfade transition should gracefully degrade on WebKitGTK (already guarded, verify it works)
- COUNTERPART_MAP resolution logic works in unit tests — runtime issue is likely state synchronization

### Claude's Discretion
- Whether to revert all 7 commits and rebuild from scratch or surgically fix each
- Optimal debounce timing for wallbash events
- Whether gsettings monitor needs a keepalive/restart mechanism
- Test strategy for verifying wallbash integration (mock events vs E2E)

</decisions>
