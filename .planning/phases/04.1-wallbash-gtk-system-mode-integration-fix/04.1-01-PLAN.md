export function setWallbashState(update: {

  colorScheme?: 'prefer-dark' | 'prefer-light'
  gtkThemeName?: string

  if (update.colors && Object.keys(update.colors).length > 0) {

}

Fix two problems in the Rust backend event emitters:

**Problem A: Wallbash watcher debounce timing**

The current watcher uses a 250ms debounce + 150ms sleep = 400ms minimum delay. The 150ms sleep is intended to wait for the second file (colors.conf or theme.conf) to finish writing. However, the debounce check `if now.duration_since(last_emit_ts) < debounce { continue; }` means events within 250ms of the last emit are silently dropped. If wallbash writes both files within 250ms (which it does), only the first one triggers an emit, and the second file may not have been written yet.

Fix: Replace the "sleep + drain" pattern with a proper coalescing debounce. Instead of sleeping 150ms inline, use a timer-based approach:

In `start_wallbash_watcher`:
1. Change the debounce from 250ms to 200ms
2. After receiving a relevant event, set a "pending emit" flag with a timestamp
3. On subsequent events within the debounce window, just update the timestamp (extend the window)
4. When no event has arrived for 200ms, read both files and emit

Implementation approach using recv_timeout:
```rust
// Replace the existing loop with:
let debounce = std::time::Duration::from_millis(200);
let mut pending = false;

loop {
    let timeout = if pending { debounce } else { std::time::Duration::from_secs(86400) };
    match rx.recv_timeout(timeout) {
        Ok(Ok(event)) => {
            use notify::EventKind;
            match event.kind {
                EventKind::Modify(_) | EventKind::Create(_) => {}
                _ => continue,
            }
            let dominated = event.paths.iter().any(|p| {
                let name = p.file_name().and_then(|n| n.to_str()).unwrap_or("");
                name == "colors.conf" || name == "theme.conf"
            });
            if dominated {
                pending = true;
            }
        }
        Ok(Err(e)) => {
            tracing::warn!("File watcher error: {}", e);
        }
        Err(std::sync::mpsc::RecvTimeoutError::Timeout) => {
            if pending {
                pending = false;
                // Read both files and emit combined event
                let colors = read_wallbash_colors_inner().ok();
                let theme = read_theme_conf_inner().ok();
                let payload = serde_json::json!({
                    "colors": colors.unwrap_or_else(|| serde_json::json!({})),
                    "theme": theme.unwrap_or_else(|| serde_json::json!({
                        "gtk_theme": "", "icon_theme": "", "color_scheme": ""
                    })),
                });
                tracing::info!("Wallbash theme update, emitting combined event");
                let _ = handle.emit("wallbash-theme-update", &payload);
            }
        }
        Err(std::sync::mpsc::RecvTimeoutError::Disconnected) => {
            tracing::info!("Wallbash watcher channel closed, stopping");
            break;
        }
    }
}
```

This eliminates the thread::sleep(150ms) and replaces the drop-based debounce with a proper coalescing debounce using recv_timeout. Events within 200ms of each other are coalesced, and the emit fires 200ms after the LAST event in a burst.

**Problem B: gsettings monitor has no restart logic**

The current `start_color_scheme_monitor` spawns a `gsettings monitor` subprocess but if it crashes or exits (which can happen on Wayland when the dbus session restarts), the monitor never comes back.

Add a restart loop with exponential backoff:
```rust
// Inside the spawn_blocking closure, wrap the existing logic in a loop:
let mut restart_delay = std::time::Duration::from_secs(1);
let max_delay = std::time::Duration::from_secs(60);

loop {
    // ... existing spawn + read logic ...

    // After the reader loop exits (subprocess died):
    tracing::warn!("gsettings monitor exited, restarting in {:?}", restart_delay);
    std::thread::sleep(restart_delay);
    restart_delay = std::cmp::min(restart_delay * 2, max_delay);
}
```

Reset the delay back to 1s after a successful run of at least 30 seconds (meaning the subprocess stayed alive long enough to be useful).

Also add `let _ = child.wait();` after the reader loop to properly reap the child process before restarting.
