<objective>
Add gateway WS proxy routes for `skills.status` and `skills.bins` so the backend fetches skills and tools data through the persistent WebSocket connection instead of the HTTP REST proxy.

Purpose: The current `GET /api/openclaw/tools` and `GET /api/openclaw/skills` in `openclaw_data.rs` use `gateway_forward()` (HTTP REST proxy). The OpenClaw protocol v3 does NOT have a `tools.list` method. The correct methods are `skills.status` (returns installed skills with their status) and `skills.bins` (returns available skill binaries/tools). The tools concept in our UI maps to `skills.bins` (available installable binaries). (RPC-08)

Output: Two new endpoints in gateway.rs (`/api/gateway/skills/status` and `/api/gateway/skills/bins`) that call the real WS RPC methods. The old listing HTTP proxy routes are removed. The `invoke_tool` HTTP proxy stays because tool invocation is a REST-specific operation.
</objective>

```rust
/// `GET /api/gateway/skills/status`
///
/// Proxies `skills.status` through the persistent gateway WS connection.
/// Returns installed skills and their current status.
async fn gateway_skills_status(
    State(state): State<AppState>,
    RequireAuth(_session): RequireAuth,
) -> Result<Json<Value>, AppError> {
    let gw = state.gateway_ws.as_ref().ok_or_else(|| {
        AppError::BadRequest(
            "OpenClaw Gateway not configured. Set OPENCLAW_WS in Settings > Connections.".into(),
        )
    })?;

    let payload = gw
        .request("skills.status", json!({}))
        .await
        .map_err(|e| {
            tracing::error!("[gateway] skills.status failed: {e}");
            AppError::BadRequest(format!("Gateway error: {}", sanitize_error_body(&e)))
        })?;

    Ok(Json(json!({
        "ok": true,
        "data": payload,
    })))
}
```
