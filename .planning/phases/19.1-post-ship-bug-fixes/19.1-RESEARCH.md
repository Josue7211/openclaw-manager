## Summary

This phase addresses 11+ bugs discovered during manual testing after v0.0.3 ship. The bugs fall into three categories: (1) UI/navigation issues (Chat tabs, Dashboard layout, duplicate pages), (2) service connectivity issues (OpenClaw health check, Models/Usage/Tools "not configured" false positive), and (3) architecture corrections (Remote Viewer noVNC->Moonlight, OpenClaw VM vs Services VM). One mysterious "ffir" error exists only in user reports -- it is NOT in the codebase, suggesting a runtime/system-level issue.

The codebase is well-structured with clear patterns. Most fixes are surgical: remove a tab, fix a health check gate, adjust a secret key lookup, redirect routes. The largest change is the Remote Viewer (Bug 6/7), which requires replacing noVNC with either a Moonlight web streaming embed or a simplified launch/status page.

**Primary recommendation:** Fix bugs in dependency order -- start with the `ffir` mystery (may affect everything), then health/config fixes (unblocks OpenClaw tabs), then UI cleanup (Chat tabs, Dashboard, dedup Agents), then Remote Viewer architecture change, and finally koel.rs commit.

## Bug Analysis

### Bug 1: "Executable not found: ffir" -- MYSTERY

**What was found:** The string "ffir" does NOT exist anywhere in the codebase (src-tauri, frontend, config files). Searched all `.rs`, `.ts`, `.tsx`, `.json` files. No Tauri sidecar config references it. No Command::new() call constructs it.

**Possible explanations (ranked by likelihood):**
1. **External tool or system service** -- something outside the app (e.g., a shell plugin, desktop integration, or Hyprland script) is firing when pages load
2. **Tauri shell plugin** -- the `tauri-plugin-shell` is enabled with `shell:allow-open`; something might be trying to open a URL scheme that triggers an external binary lookup
3. **ffmpeg typo** -- the messages route uses `ffmpeg` for video transcoding; a garbled binary name could theoretically appear, but this seems unlikely
4. **Cached error** -- a stale build artifact or cached response from a previous version

**Confidence:** LOW -- cannot determine root cause from code alone. Needs runtime debugging.

**Recommended investigation approach:**
- Check if error appears in Tauri dev console (F12 in the webview)
- Search system PATH for any binary containing "ffir": `which ffir; find /usr -name "ffir*"`
- Check Hyprland/Waybar configs for shell commands triggered by window focus
- Add a global error interceptor temporarily to capture the full error stack trace
- Check if the Axum server logs show any `ffir` references at startup

### Bug 2: Chat "Bjorn" Tab (CONFIRMED)

**Root cause:** `frontend/src/pages/Chat.tsx` has a two-tab switcher (lines 21, 61-89). Tab state `useState<'chat' | 'bjorn'>('chat')` creates two views: the standard ChatThread and a lazy-loaded BjornTab.

**The Bjorn tab** (`BjornTab.tsx`) is a separate module-builder experience where users describe modules and Bjorn generates React components with preview. This is a distinct feature from regular Chat.

**Fix:** The user says "Chat IS Bjorn, one tab only." Two approaches:
1. **Merge**: Keep BjornTab functionality accessible but remove the tab switcher. Add a toggle or mode switch within the single chat interface.
2. **Remove tab**: Simply remove the tab UI, default to 'chat' tab, and keep BjornTab accessible via a different entry point (or remove entirely).

**Recommendation:** Remove the tab switcher entirely. Keep the ChatThread as the sole Chat page content. The BjornTab component can stay in the codebase (it's a separate feature) but remove its entry point from Chat.tsx. If Bjorn module building is wanted later, it can be a separate page.

**Files to modify:** `frontend/src/pages/Chat.tsx`

### Bug 3: Dashboard -- Only 1 Widget in Browser Mode (CONFIRMED)

**Root cause analysis:** The dashboard uses `localStorage` (`dashboard-state` key) for persistence via `dashboard-store.ts`. On first use, `createInitialState()` creates a page with empty `layouts: {}`. The `useEffect` in `Dashboard.tsx` (lines 50-64) detects empty layouts and calls `generateDefaultLayout()` to populate all 8 default widgets.

**The browser mode issue** is likely one of:
1. **Different localStorage namespace** -- Tauri webview and browser at localhost:5173 share different localStorage. If the browser had an old/partial state from testing, it would use that instead of generating defaults.
2. **Stale localStorage** -- A previous session stored a dashboard state with only 1 widget, which persists.
3. **Layout key mismatch** -- The `useEffect` only triggers when `Object.keys(activePage.layouts).length === 0`. If layouts exist but with only 1 widget, defaults are not regenerated.

**The real bug** is likely #3: if a user or previous code version saved a partial layout (1 widget), the `layouts` object is non-empty, so the effect never fires to populate defaults. The check should be more robust.

**Fix:** Add a migration or check that detects suspiciously small layouts (e.g., < 3 widgets) and offers to reset to defaults, OR fix the initial state logic to always generate defaults when total widget count is below a threshold.

**Files to modify:** `frontend/src/pages/Dashboard.tsx` (first-use effect), possibly `frontend/src/lib/dashboard-defaults.ts`

### Bug 4 & 5: OpenClaw Tabs "Not Configured" / Empty (CONFIRMED -- SAME ROOT CAUSE)

**Root cause:** The OpenClaw health check in `gateway.rs` (line 174-200) looks up `OPENCLAW_API_URL` via `state.secret("OPENCLAW_API_URL")`. If this returns `None`, health returns `{ "ok": false, "status": "not_configured" }`. The frontend health query in `OpenClaw.tsx` (line 358-363) gates all tab content on `healthy = healthData?.ok ?? false`.

When health returns `ok: false`, all tabs show "not configured" messages because:
- `ModelsTab.tsx` checks `if (!healthy)` and returns the "not configured" banner
- `UsageTab.tsx` same pattern
- `ToolsTab.tsx` same pattern
- `AgentsTabContent` doesn't check health for data loading, but `useAgents` fetches from `/api/agents` which itself may fail

**Browser mode specifics:** In browser mode (localhost:5173), the fetch wrapper in `lib/api.ts` sends requests to `http://127.0.0.1:3000`. In debug mode, the Axum server allows requests from localhost without API key (per CLAUDE.md). So auth should work. The issue is more likely that:
1. The user hasn't configured `OPENCLAW_API_URL` in the OS keychain/secrets store
2. The secret is stored but the health check URL (`{base}/health`) is pointing at the wrong host (OpenClaw VM instead of Services VM)
3. The upstream health endpoint at Services VM is unreachable

**Architecture bug:** Per the user's note, OPENCLAW_API_URL should point to Services VM (10.0.0.109), NOT OpenClaw VM (10.0.0.173). If the URL was configured pointing at OpenClaw VM (which only runs Moonlight/Sunshine), health would return "unreachable" which maps to `ok: false`.

**Fix:**
1. Document that OPENCLAW_API_URL must point to Services VM
2. Update CLAUDE.md infrastructure diagram
3. If needed, add a Settings hint or validation that checks which VM the URL points to

**Files to modify:** `CLAUDE.md` (infrastructure diagram), potentially `src-tauri/src/tailscale.rs` (expected host mapping), Settings connections documentation

### Bug 6 & 7: Remote Viewer -- noVNC to Moonlight/Sunshine

**Current implementation:**
- Backend: `src-tauri/src/routes/vnc.rs` -- WebSocket proxy relaying binary VNC frames to a TCP VNC server via `VNC_HOST` secret
- Frontend: `VncViewer.tsx` + `useVnc.ts` using `@novnc/novnc` (RFB protocol)
- CAS guard limits to 2 concurrent VNC sessions

**Required change:** Replace noVNC/VNC with Moonlight/Sunshine streaming.

**Moonlight/Sunshine architecture:**
- Sunshine runs on the OpenClaw VM as a game streaming host
- Moonlight is the client -- native apps exist for all platforms
- **moonlight-web-stream** is a community project that bridges Sunshine to browsers via WebRTC
  - GitHub: https://github.com/MrCreativ3001/moonlight-web-stream
  - 436 stars, active development, uses WebRTC API
  - Requires running a separate web server that proxies Sunshine traffic
  - Supports H.264, H.265, AV1 video codecs

**Options:**
1. **Embed moonlight-web-stream via iframe** -- Run moonlight-web-stream on OpenClaw VM, iframe it in the app. Simplest but requires deploying another service.
2. **Launch Moonlight native client** -- Just open the native Moonlight app with the correct connection. The Remote Viewer page becomes a status/launch page.
3. **Simplified status page** -- Show connection status to OpenClaw VM, with a "Launch Moonlight" button that opens the native app. No streaming in-browser.

**Recommendation:** Option 3 (simplified status page) for this bug-fix phase. Embedding a full streaming client is a feature, not a bug fix. The page should:
- Show Sunshine service status on OpenClaw VM (ping check)
- Provide a "Launch Moonlight" button that opens `moonlight://` URL scheme or the native app
- Optionally show the Sunshine web admin URL for configuration
- Remove the noVNC infrastructure entirely (vnc.rs, useVnc.ts, VncViewer.tsx, VncToolbar.tsx)

**Files to modify:** `src-tauri/src/routes/vnc.rs` (replace with sunshine status), `frontend/src/pages/remote/RemotePage.tsx`, `frontend/src/pages/remote/VncViewer.tsx` (remove), `frontend/src/pages/remote/VncToolbar.tsx` (remove), `frontend/src/hooks/useVnc.ts` (remove), `frontend/src/lib/modules.ts` (update requiresConfig)

## Architecture Patterns

### Service Configuration Pattern
All service URLs are stored in the OS keychain via `secrets.rs`. Route handlers access them via `state.secret("KEY_NAME")`. The pattern is consistent:
1. Check secret exists and is non-empty
2. If not configured, return a "not configured" response
3. If configured, make the upstream request

### Health Check Pattern
The gateway health check (`/api/openclaw/health`) probes `{OPENCLAW_API_URL}/health` with a 5-second timeout. Returns JSON with `ok: boolean` and `status: string`. Frontend tabs gate rendering on `ok`.

### Tab Content Gating
All OpenClaw tabs follow the same pattern: check `healthy` prop, show "not configured" banner if false, otherwise render content. This means fixing the health check automatically unblocks all tabs.

## Common Pitfalls

### Pitfall 1: Browser Mode Auth
**What goes wrong:** API calls from browser at localhost:5173 fail because there's no MC_API_KEY.
**Why it happens:** The API key is loaded from OS keychain via Tauri IPC. In browser mode, Tauri IPC is unavailable.
**How to avoid:** Debug mode already allows localhost requests without API key (confirmed in CLAUDE.md). If API calls fail in browser mode, check that the Axum server is running and accepting unauthenticated localhost requests.
**Warning signs:** 401/403 errors in browser console.

### Pitfall 2: Health Check False Negatives
**What goes wrong:** Health check returns "unreachable" when the service IS running, causing all tabs to show "not configured."
**Why it happens:** Wrong host IP, wrong port, firewall blocking, or the upstream health endpoint path is wrong.
**How to avoid:** Verify OPENCLAW_API_URL points to Services VM. Test with curl: `curl http://<services-vm-ip>:<port>/health`.
**Warning signs:** Health endpoint returns 200 with `ok: false` and `status: unreachable`.

### Pitfall 3: Stale localStorage State
**What goes wrong:** Dashboard shows wrong widget count because old localStorage state is preserved.
**Why it happens:** localStorage persists across app updates. Old state may have incompatible structure.
**How to avoid:** Either add a migration version check or make the default layout regeneration more robust.
**Warning signs:** Dashboard looks different in fresh incognito vs normal browsing.

### Pitfall 4: Axum Route Registration with Response Types
**What goes wrong:** Handlers returning `Result<Response, AppError>` silently fail to register in merged routers.
**Why it happens:** Documented Axum gotcha in CLAUDE.md.
**How to avoid:** Use `Result<Json<Value>, AppError>` for most handlers. WebSocket upgrade handlers return bare `Response`.
**Warning signs:** Routes compile but return 404 at runtime.
