# Phase 02.2: Theme System Mode Fixes - Research

**Researched:** 2026-03-20
**Domain:** Desktop theme integration (Hyprland/Wallbash + cross-platform dark mode)
**Confidence:** HIGH

## Summary

This phase fixes System mode to truly mirror the user's desktop theme. The current implementation maps a GTK theme name to a static built-in preset and polls `gsettings` every 1s. The target state is: (1) read live Wallbash colors directly from `~/.config/hypr/themes/colors.conf` when Wallbash-Gtk is active, (2) watch the file for instant sync instead of polling, (3) show only the active system theme card in the UI, (4) auto-switch presets when toggling Dark/Light modes, (5) fix light theme text contrast for real-world surfaces, and (6) degrade gracefully on Windows/macOS where GTK detection is unavailable.

The core technical challenge is building a Rust Tauri command that parses the Wallbash `colors.conf` format (4 color groups x 11 variables each, with hex and rgba variants), mapping those raw colors to the app's CSS variable system, and wiring a file watcher through Tauri events to the frontend. The codebase already has the full theme engine infrastructure; this phase extends it rather than replacing it.

**Primary recommendation:** Use the `notify` crate (v8.x) for file watching in Rust, emit Tauri events to the frontend on file changes, and add a new `WallbashThemeDefinition` type that resolves live colors instead of static presets.

### Pattern 1: Rust File Watcher with Tauri Events

**What:** A background `notify` watcher monitors `colors.conf` and emits Tauri events when the file changes.
**When to use:** Linux only (Wallbash is Hyprland-specific). The watcher only starts if the file exists and the platform is Linux.

```rust
// In main.rs .setup(), after server spawn:
#[cfg(target_os = "linux")]
{
    let handle_clone = handle.clone();
    tauri::async_runtime::spawn(async move {
        crate::commands::start_wallbash_watcher(handle_clone).await;
    });
}
```

```rust
// In commands.rs:
use notify::{Watcher, RecursiveMode, Event, EventKind};
use std::sync::mpsc;

pub async fn start_wallbash_watcher(handle: tauri::AppHandle) {
    let colors_path = dirs::home_dir()
        .map(|h| h.join(".config/hypr/themes/colors.conf"));
    let theme_path = dirs::home_dir()
        .map(|h| h.join(".config/hypr/themes/theme.conf"));

    let Some(colors_path) = colors_path else { return };
    if !colors_path.exists() { return }

    let (tx, rx) = mpsc::channel();
    let mut watcher = match notify::recommended_watcher(tx) {
        Ok(w) => w,
        Err(_) => return,
    };

    // Watch the parent directory (more reliable than individual files)
    let parent = colors_path.parent().unwrap();
    if watcher.watch(parent, RecursiveMode::NonRecursive).is_err() {
        return;
    }

    loop {
        match rx.recv() {
            Ok(Ok(Event { kind: EventKind::Modify(_) | EventKind::Create(_), paths, .. })) => {
                // Only react to changes in our target files
                let relevant = paths.iter().any(|p| {
                    p.ends_with("colors.conf") || p.ends_with("theme.conf")
                });
                if relevant {
                    if let Ok(colors) = read_wallbash_colors_inner() {
                        let _ = handle.emit("wallbash-colors-changed", &colors);
                    }
                    if let Ok(theme) = read_theme_conf_inner() {
                        let _ = handle.emit("gtk-theme-changed", &theme);
                    }
                }
            }
            Ok(Err(_)) | Err(_) => break,
        }
    }
}
```

### Pattern 2: Wallbash Color Parsing

**What:** Parse the `colors.conf` hyprlang format into a structured color map.
**Format:** Each color group has: `$wallbash_pry{N} = RRGGBB`, `$wallbash_txt{N} = RRGGBB`, and 9 accent tiers `$wallbash_{N}xa{1-9} = RRGGBB`. There are also `_rgba` variants which can be ignored (we compute our own alpha values).

```rust
// Tauri command
#[tauri::command]
pub fn read_wallbash_colors() -> Result<serde_json::Value, String> {
    read_wallbash_colors_inner().map_err(|e| e.to_string())
}

fn read_wallbash_colors_inner() -> Result<serde_json::Value, std::io::Error> {
    let path = dirs::home_dir()
        .ok_or_else(|| std::io::Error::new(std::io::ErrorKind::NotFound, "no home dir"))?
        .join(".config/hypr/themes/colors.conf");
    let content = std::fs::read_to_string(&path)?;
    let mut colors = serde_json::Map::new();

    for line in content.lines() {
        let line = line.trim();
        if line.starts_with('#') || line.is_empty() { continue }
        // Skip rgba variants — we only need hex
        if line.contains("_rgba") { continue }
        // Parse: $wallbash_pry1 = 0A060D    # comment
        if let Some(rest) = line.strip_prefix('$') {
            if let Some((name, value_comment)) = rest.split_once('=') {
                let name = name.trim();
                let value = value_comment.split('#').next().unwrap_or("").trim();
                if value.len() == 6 && value.chars().all(|c| c.is_ascii_hexdigit()) {
                    colors.insert(name.to_string(), serde_json::Value::String(format!("#{}", value)));
                }
            }
        }
    }

    Ok(serde_json::Value::Object(colors))
}
```

### Pattern 3: Wallbash Colors to CSS Variable Mapping

**What:** Map the 4 wallbash color groups to the app's CSS variable space.
**Mapping (Claude's discretion area):**

| Wallbash Variable | CSS Variable | Rationale |
|-------------------|-------------|-----------|
| `wallbash_pry1` | `--bg-base` | Darkest primary = base background |
| `wallbash_pry2` | `--bg-panel` (solid), `--bg-card-solid` | Second darkest = panels/cards |
| `wallbash_pry3` | `--bg-elevated` (solid), `--bg-popover` | Mid-tone = elevated surfaces |
| `wallbash_pry4` | Used for hover/active calculations | Lightest primary, used derivatively |
| `wallbash_txt1` | `--text-primary` | Primary text on darkest bg |
| `wallbash_txt4` | `--text-primary` (light mode) | Text for lightest surfaces |
| `wallbash_3xa5` | `--accent` | Mid-tier accent = main accent color |
| `wallbash_3xa3` | `--accent-dim` | Lower accent tier |
| `wallbash_3xa7` | `--accent-bright` | Higher accent tier |
| `wallbash_1xa5` | `--secondary` (green/status) | Group 1 mid accent |
| `wallbash_4xa5` | `--tertiary` (blue/dashboard) | Group 4 mid accent |
| `wallbash_3xa5_rgba` values | `--glow-top-rgb` | Glow from accent group |

**Dark vs Light detection:** Read `$COLOR_SCHEME` from `theme.conf`. If `prefer-dark`, use groups 1-2 for backgrounds and txt1/txt2 for text. If `prefer-light`, invert: use groups 3-4 for backgrounds and txt3/txt4 for text. The app does NOT switch its mode toggle -- it just applies the colors the desktop dictates.

```typescript
// In theme-engine.ts
interface WallbashColors {
  [key: string]: string // e.g. "wallbash_pry1" -> "#11151A"
}

export function buildWallbashTheme(
  colors: WallbashColors,
  colorScheme: 'prefer-dark' | 'prefer-light'
): ThemeDefinition {
  const isDark = colorScheme === 'prefer-dark'

  // Select background groups based on dark/light
  const bgBase = isDark ? colors.wallbash_pry1 : colors.wallbash_pry4
  const bgPanel = isDark ? colors.wallbash_pry2 : colors.wallbash_pry3
  const textPrimary = isDark ? colors.wallbash_txt1 : colors.wallbash_txt4
  const accent = colors.wallbash_3xa5

  return {
    id: 'wallbash-live',
    name: 'Wallbash',
    category: isDark ? 'dark' : 'light',
    builtIn: true,
    colors: {
      'bg-base': bgBase,
      'bg-panel': `rgba(${hexToRgbStr(bgPanel)}, 0.85)`,
      'bg-card-solid': bgPanel,
      // ... full mapping
      'text-primary': textPrimary,
      'accent': accent,
      // ... etc
    }
  }
}
```

### Pattern 4: Counterpart Auto-Switch

**What:** When user toggles Dark<->Light mode, auto-switch to the counterpart theme preset.

```typescript
// In theme-definitions.ts
export const COUNTERPART_MAP: Record<string, string> = {
  'default-dark': 'default-light',
  'default-light': 'default-dark',
  'gruvbox-dark': 'gruvbox-light',
  'gruvbox-light': 'gruvbox-dark',
  'catppuccin-mocha': 'catppuccin-latte',
  'catppuccin-latte': 'catppuccin-mocha',
  'solarized-dark': 'solarized-light',
  'solarized-light': 'solarized-dark',
  'high-contrast-dark': 'high-contrast-light',
  'high-contrast-light': 'high-contrast-dark',
}

// Themes without counterpart: dracula, nord, rose-pine, tokyo-night,
// graphite-mono, decay-green, edge-runner, synth-wave, wallbash,
// material-sakura, terminal, purple-mode, pink-mode, monster-high
// These fall back to default-dark/default-light.
```

```typescript
// In theme-store.ts, modify setMode():
export function setMode(mode: 'dark' | 'light' | 'system') {
  const prevMode = _state.mode
  const prevThemeId = _state.activeThemeId

  // Auto-switch to counterpart when toggling dark<->light
  let newThemeId = prevThemeId
  if (prevMode !== mode && mode !== 'system' && prevMode !== 'system') {
    const counterpart = COUNTERPART_MAP[prevThemeId]
    if (counterpart) {
      newThemeId = counterpart
    } else {
      // No counterpart — fall back to default for that mode
      newThemeId = mode === 'dark' ? 'default-dark' : 'default-light'
    }
  }

  mutate(s => ({ ...s, mode, activeThemeId: newThemeId }))
  applyThemeFromState()
}
```

### Pattern 5: Frontend Event Listener (Replace Polling)

**What:** Replace the 1s `setInterval` polling in `main.tsx` with Tauri event listeners.

```typescript
// In main.tsx, replace the setInterval block:
import { listen } from '@tauri-apps/api/event'

// File watcher events from Rust (instant, replaces 1s polling)
listen('wallbash-colors-changed', (event) => {
  if (getThemeState().mode !== 'system') return
  const colors = event.payload as WallbashColors
  setWallbashColors(colors) // Store in theme-engine module state
  applyThemeFromState()
})

listen('gtk-theme-changed', (event) => {
  if (getThemeState().mode !== 'system') return
  const themeInfo = event.payload as { gtk_theme: string; color_scheme: string }
  setGtkThemeMapping(themeInfo.gtk_theme)
  setOsDarkPreference(themeInfo.color_scheme === 'prefer-dark')
  applyThemeFromState()
})

// Keep gsettings polling as fallback for non-Wallbash GTK themes
// (Adwaita, etc. don't write to colors.conf)
// But reduce interval to 3s since file watcher handles the fast path
```

### Anti-Patterns to Avoid
- **Polling for Wallbash colors:** The file watcher provides instant sync. Only use polling as a fallback when `notify` fails to initialize.
- **Switching app mode on Wallbash dark/light change:** The CONTEXT.md is explicit: Wallbash dark/light changes update CSS variables but do NOT flip the app's mode toggle. The app mirrors the desktop.
- **Building a full ThemeDefinition from scratch each time:** Reuse the existing `applyTheme()` pipeline. Build a ThemeDefinition object from wallbash colors and feed it through the same path.
- **Parsing rgba variants from colors.conf:** The hex values are sufficient. The app computes its own rgba alpha tints via `deriveAlphaTints()` etc.

## Common Pitfalls

### Pitfall 1: File Watcher Race Condition
**What goes wrong:** Wallbash writes `colors.conf` atomically but the watcher fires on the temp file creation, not the final rename. You read partial/empty content.
**Why it happens:** `notify` fires events for every filesystem operation (create, write, rename). A typical atomic write is: write to `.colors.conf.tmp` -> rename to `colors.conf`.
**How to avoid:** Debounce the watcher events. Only react after a 50-100ms quiet period. Or check that the event path ends with `colors.conf` (not a temp file).
**Warning signs:** Intermittent empty/corrupt color reads. Theme flickers to defaults then corrects.

### Pitfall 2: Wallbash Light Mode Text on Dark Backgrounds
**What goes wrong:** The wallbash color file has `txt1 = FFFFFF` (white text) for groups 1-2 but `txt4 = 101111` (dark text) for group 4. If the mapping assumes all text is white, light mode becomes unreadable.
**Why it happens:** Each color group is designed for its own primary background. Group 1 (dark bg) needs light text; group 4 (light bg) needs dark text.
**How to avoid:** Always pair `txt{N}` with `pry{N}`. When `COLOR_SCHEME` is `prefer-light`, use group 4's text colors.
**Warning signs:** White text on light backgrounds, or dark text on dark backgrounds.

### Pitfall 3: System Mode Card Still Showing All Themes
**What goes wrong:** The `filteredThemes` logic in both `ThemePicker.tsx` and `SettingsDisplay.tsx` currently shows ALL themes when `mode === 'system'`. Both files need separate system-mode rendering paths.
**Why it happens:** Two independent components implement theme filtering. Easy to fix one and miss the other.
**How to avoid:** Extract the system-mode filtering logic into a shared hook or utility. Update both `ThemePicker.tsx` and `SettingsDisplay.tsx` together.
**Warning signs:** Settings shows single card but ThemePicker still shows grid (or vice versa).

### Pitfall 4: Contrast Tests Pass but Users Report Unreadable Text
**What goes wrong:** The 120 existing tests check `text-primary/secondary/muted` against `bg-base` and `bg-card-solid`. But text also appears on `bg-panel`, `bg-elevated`, `bg-popover`, and various alpha-blended surfaces that aren't tested.
**Why it happens:** Alpha-blended backgrounds (e.g., `rgba(255, 255, 255, 0.85)` on a light bg-base) composite to near-white, reducing contrast with light text-secondary.
**How to avoid:** Add contrast tests for text on composited surfaces: resolve `bg-panel` rgba against `bg-base`, then test text against the composited result.
**Warning signs:** All automated tests green but visual inspection shows low-contrast text.

### Pitfall 5: notify Crate Blocking Main Thread
**What goes wrong:** The `notify` watcher's `recv()` is blocking. If placed on the main Tauri thread, it freezes the app.
**Why it happens:** `notify`'s channel receiver blocks the calling thread until an event arrives.
**How to avoid:** Always run the watcher in `tauri::async_runtime::spawn()` (which uses tokio). Use `tokio::task::spawn_blocking()` for the blocking `recv()` loop.
**Warning signs:** App hangs on startup, especially when colors.conf doesn't change frequently.

### Pitfall 6: Windows/macOS System Mode Shows "System Theme" Card
**What goes wrong:** The "System Theme: Wallbash" card concept is Linux-specific. On Windows/macOS, there's no GTK theme to display.
**Why it happens:** Conditional logic for platform detection gets complex. Easy to show the wrong UI.
**How to avoid:** Check `navigator.userAgent.includes('Linux')` on the frontend. The CONTEXT.md specifies: Windows/macOS shows filtered presets, not a single system card.
**Warning signs:** macOS users see a blank "System Theme" card with no colors.

## Open Questions

1. **What happens when the wallpaper changes but colors.conf is regenerated slowly?**
   - What we know: Wallbash writes colors.conf atomically after wallpaper analysis completes.
   - What's unclear: The exact delay between wallpaper change and colors.conf update.
   - Recommendation: The file watcher handles this naturally -- it fires when the file is written. Add a debounce of 100ms to avoid partial reads during atomic write sequences.

2. **Should the wallbash live theme support user color overrides?**
   - What we know: CONTEXT.md says "User can still customize accent/secondary/tertiary colors on top of the system theme."
   - What's unclear: How overrides interact with live color updates. If user overrides accent, and wallbash regenerates, does the override persist or get overwritten?
   - Recommendation: User overrides should persist. Store wallbash overrides under a `wallbash-live` key in `state.overrides`. When wallbash updates, re-apply base colors but keep user overrides on top.

3. **How to handle transition between Wallbash and non-Wallbash GTK themes in system mode?**
   - What we know: User could switch from Wallbash-Gtk to Rose-Pine via HyDE theme switcher.
   - What's unclear: The transition UX.
   - Recommendation: `theme.conf` also changes when GTK theme switches. The watcher picks up both files. If GTK_THEME is no longer Wallbash-Gtk, fall back to the static preset mapping (existing behavior).
