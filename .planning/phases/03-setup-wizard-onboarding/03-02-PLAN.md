Create `src-tauri/src/routes/wizard.rs` with:

1. **WizardTestRequest struct:**
   ```rust
   #[derive(Deserialize)]
   struct WizardTestRequest {
       service: String,  // "supabase" | "openclaw" | "bluebubbles" | "couchdb" | "mac-bridge"
       url: String,
       #[serde(default)]
       key: String,      // API key, anon key, or password depending on service
       #[serde(default)]
       password: String,
       #[serde(default)]
       username: String,
   }
   ```

2. **POST /wizard/test-connection** handler (`wizard_test_connection`):
   - Uses `State(state): State<AppState>` for the HTTP client (`state.http` or create a fresh reqwest client with 5-second timeout)
   - Match on `body.service`:
     - `"supabase"`: GET `{url}/rest/v1/` with `apikey` and `Authorization: Bearer {key}` headers. 200-399 = ok, 401 = auth error, else = connection error.
     - `"openclaw"`: GET `{url}/health` or `{url}/v1/models` with `Authorization: Bearer {key}`. Check for 200.
     - `"bluebubbles"`: GET `{url}/api/v1/server/info?password={password}`. Check for 200.
     - `"couchdb"`: GET `{url}/` with Basic auth (username:password). Check for 200.
     - `"mac-bridge"`: GET `{url}/health` with `X-API-Key: {key}` header. Check for 200.
   - Return JSON: `{ "status": "ok", "latency_ms": N }` on success, `{ "status": "error", "error": "message" }` on failure
   - Error messages follow UI-SPEC patterns: "Connection refused on port {port} -- is {service} running?", "Authentication failed -- check your API key", "Connection timed out -- check that {service} is reachable", "Could not resolve hostname -- check the URL"
   - If URL contains "100." prefix and connection fails, append hint: "This looks like a Tailscale address -- is Tailscale connected?"
   - Log test attempts using `tracing::info!` with redacted URLs (use `redact()` from redact.rs)
   - Rate-limit: at most 10 requests per minute per service (simple in-memory counter or accept the existing rate limiter)

3. **WizardSaveRequest struct:**
   ```rust
   #[derive(Deserialize)]
   struct WizardSaveRequest {
       credentials: HashMap<String, String>,  // keychain key -> value pairs
   }
   ```

4. **POST /wizard/save-credentials** handler:
   - Iterates `body.credentials` and calls `set_entry(key, value)` for each
   - Only allows keys that exist in KEY_ENV_MAP (validate against allowed keys)
   - Returns `{ "saved": N }` count or error
   - Logs "Wizard saved {N} credentials" (no credential values logged)

5. **POST /wizard/reload-secrets** handler:
   - Calls `load_secrets()` to re-read all keychain values
   - Updates `AppState.secrets` (the secrets HashMap) via the write lock mechanism
   - Check how AppState stores secrets -- if it's a `RwLock<HashMap>`, write-lock and replace. If it's a plain HashMap set at startup, you may need to add a `RwLock` wrapper or `ArcSwap`. Look at AppState definition in server.rs to determine the exact approach.
   - Returns `{ "reloaded": true }`

6. **Router function:**
   ```rust
   pub fn router() -> Router<AppState> {
       Router::new()
           .route("/wizard/test-connection", post(wizard_test_connection))
           .route("/wizard/save-credentials", post(wizard_save_credentials))
           .route("/wizard/reload-secrets", post(wizard_reload_secrets))
   }
   ```

7. **Register in mod.rs:** Add `pub mod wizard;` declaration and `.merge(wizard::router())` in the main router function.

8. **Auth exemption in server.rs:** Add `"/api/wizard/"` to `AUTH_EXEMPT_PREFIXES` array. This ensures wizard endpoints only need X-API-Key (which the frontend has via OS keychain), NOT a user session.

9. **Include `#[cfg(test)] mod tests` block** with at least:
   - Test that WizardTestRequest deserializes correctly
