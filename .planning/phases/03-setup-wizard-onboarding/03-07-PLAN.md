1. In `src-tauri/src/commands.rs`, the `start_wallbash_watcher` function currently checks `dominated` (whether colors.conf or theme.conf is in the event paths). The coalesce logic (150ms sleep + drain) is correct, but needs to ensure theme.conf-only changes still emit.

   **Verify the current watcher fires on theme.conf-only changes.** The `dominated` check already includes theme.conf:
   ```rust
   let dominated = event.paths.iter().any(|p| {
       let name = p.file_name().and_then(|n| n.to_str()).unwrap_or("");
       name == "colors.conf" || name == "theme.conf"
   });
   ```

   This should already fire. The real issue is likely on the frontend side.

2. In `frontend/src/lib/theme-engine.ts`, the issue is that `setWallbashColors()` compares by reference — if the same colors object comes in, the state doesn't "change" from the frontend's perspective. But `setWallbashColorScheme()` DOES update the scheme. The problem: `applyThemeFromState()` in main.tsx calls `resolveThemeDefinition()` which calls `buildWallbashTheme(_wallbashColors, _wallbashColorScheme)`. If the scheme changed, this SHOULD return a different theme (different category, different bg-base from pry1 vs pry4).

   **The actual bug:** In `main.tsx` line ~141, the `wallbash-theme-update` listener checks `if (getThemeState().mode !== 'system') return`. This is correct. But after setting colors + scheme + calling `applyThemeFromState()`, the theme engine may cache or skip if it thinks nothing changed.

   Check if `applyThemeFromState()` in theme-store.ts has any caching that would prevent re-application:
   - Read `frontend/src/lib/theme-store.ts` — find `applyThemeFromState` and check if it short-circuits
   - Read `frontend/src/lib/themes.ts` — find the actual CSS variable application function

   **Fix approach:** Add a `_wallbashGeneration` counter that increments on every wallbash update. Use this in the theme resolution to bust any caching. In `theme-engine.ts`:
   ```typescript
   let _wallbashGeneration = 0

   export function setWallbashColors(colors: WallbashColors) {
     _wallbashColors = colors
     _wallbashLastUpdate = Date.now()
     _wallbashGeneration++
   }

   export function setWallbashColorScheme(scheme: 'prefer-dark' | 'prefer-light') {
     _wallbashColorScheme = scheme
     _wallbashLastUpdate = Date.now()
     _wallbashGeneration++
   }

   export function getWallbashGeneration(): number {
     return _wallbashGeneration
   }
   ```

   Then in whatever applies the theme, ensure it doesn't skip when generation changed. If `applyThemeFromState` in theme-store.ts has a check like "skip if same theme ID", the wallbash-live ID is always the same — so it skips. Fix by adding generation to the comparison or by forcing re-apply.

   In `frontend/src/lib/themes.ts` (or wherever `applyTheme` lives), find the skip condition and ensure wallbash always re-applies. The simplest fix: in `applyThemeFromState`, if the resolved theme is `wallbash-live`, always re-apply CSS variables (never skip).

3. Update `frontend/src/lib/__tests__/theme-wallbash.test.ts`:
   - Add test: same colors + different scheme → different bg-base
   - Add test: `_wallbashGeneration` increments on scheme change

4. Test by checking `cargo check` and `npx vitest run src/lib/__tests__/theme-wallbash.test.ts`.
