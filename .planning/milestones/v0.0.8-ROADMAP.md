## Phases

### v0.0.8 -- Release Package & Backend-First Setup

**Goal:** Ship a working desktop package backed by a simple self-hosted stack so setup is reduced to one Docker Compose deployment, one `.env`, one backend URL, and one pairing token.

**Branch plan** *(keep release work narrow and reviewable)*
- `chore/release-roadmap-ops` - release checklist, acceptance criteria, branch freeze, PR queue
- `feature/backend-stack-compose` - Docker Compose stack, env contract, service health wiring
- `feature/setup-status-contract` - pairing token bootstrap, setup status API, capability discovery
- `feature/onboarding-server-first` - desktop setup flow rewrite and recovery UX
- `feature/release-package-hardening` - package smoke tests, offline degradation, docs, release candidate

**PR order:**
1. Release branch discipline and acceptance criteria
2. Compose stack and env contract
3. Setup status and pairing API
4. Desktop onboarding rewrite
5. Package hardening and release candidate verification

**Group AO: Release Operations** *(stop drift and define what "working package" means)*
- [ ] **Phase 111: Release Branch & Package Plan** - Freeze the release lane, define package acceptance criteria, and document the merge order

**Group AP: Backend Stack** *(one compose file, one env file)*
- [ ] **Phase 112: Docker Compose Backend Stack** - Create a backend stack for Supabase, OpenClaw, MemD, secrets, and reverse-proxy/service glue
- [ ] **Phase 113: Unified Environment Contract** - Define a single `.env.example`, document which services consume which vars, and remove one-off manual setup steps

**Group AQ: Pairing & Setup Contract** *(paste URL, paste token, done)*
- [ ] **Phase 114: Setup Status & Pairing API** - Add a backend contract that reports health, enabled providers, missing config, and pairing state
- [ ] **Phase 115: Server-First Onboarding Flow** - Rewrite onboarding so the desktop app starts with backend URL + pairing token and only asks follow-up questions when setup is incomplete

**Group AR: Package Hardening** *(desktop package must degrade cleanly when services are absent)*
- [ ] **Phase 116: Package Smoke & Offline Degradation** - Verify auth, chat, dashboard, and offline integration states in the packaged app with clear user-facing failures
- [ ] **Phase 117: Release Candidate Build** - Produce and validate a working package, update release docs, and close the release checklist
