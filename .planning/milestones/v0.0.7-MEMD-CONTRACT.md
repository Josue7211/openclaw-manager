# MemD Contract Draft

**Milestone:** v0.0.7
**Status:** Draft
**Purpose:** Define MemD as the AI brain layer for OpenClaw Manager. MemD is responsible for durable memory, retrieval, ranking, and compaction. AgentShell and agent secrets sit around it as safety and capability layers, not as the memory substrate itself.

## Design Goals

- Keep the brain layer explicit and versioned.
- Separate durable memory from secret storage and transport adapters.
- Make retrieval deterministic enough to test, but flexible enough to evolve.
- Allow the product to grow memory-aware features without turning every page into a custom cache.

## Core Concepts

### Memory item

A memory item is a versioned unit of durable context. Examples:

- user preference
- recurring task or workflow note
- session summary
- project fact
- long-lived agent instruction

Each memory item should carry:

- stable ID
- scope
- kind
- payload
- source
- timestamps
- confidence / priority
- retention policy
- audit trail metadata

### Scope

MemD must support explicit scopes so memory can be isolated by:

- user
- session
- workspace
- agent
- integration

Scoped memory is mandatory. Global memory is allowed only for non-sensitive, system-level defaults.

### Retrieval context

MemD should expose a curated retrieval result rather than raw store access. Retrieval results should include:

- matched item IDs
- redacted payload summaries
- relevance score or rank
- reason for inclusion
- retention hints

The frontend should not receive unrestricted access to the underlying store.

## Required Behaviors

### Write path

- All writes are versioned.
- Writes are attributable to a source action.
- Sensitive content is redacted before logs or telemetry-like traces.
- Conflicting writes should be resolved by explicit policy, not accidental last-write-wins behavior.

### Read path

- Reads are filtered by scope and permission.
- Retrieval should prefer recent, relevant, and high-confidence items.
- The brain layer should support session bootstrap context and ongoing conversation context.

### Compaction path

- MemD must support pruning stale items.
- Compaction should preserve meaning, not just delete data.
- Compaction runs should produce a summary record so the system can explain what changed.

## Suggested API Shape

This is the suggested backend contract for the first branch.

- `POST /api/memd/query`
- `POST /api/memd/upsert`
- `POST /api/memd/archive`
- `POST /api/memd/compact`
- `GET /api/memd/scopes`
- `GET /api/memd/audit`

The first implementation can map these to backend routes or internal commands, but the contract should remain stable once the branch lands.

## UI Surfaces

- Memory review panel
- Scoped memory browser
- Compact / prune confirmation flow
- Memory provenance details

These should be introduced only after the backend contract is stable.

## Safety Rules

- Never store raw secrets in MemD.
- Never leak memory contents into browser globals.
- Never treat AgentShell as the memory authority.
- Never let memory retrieval bypass existing auth or scope checks.
- Never compact without keeping an audit record.

## Branch Plan

1. `feature/memd-brain`
2. `feature/agentshell-contract`
3. `feature/agent-secrets`
4. `feature/agent-platform-integration`

## Acceptance Criteria

- MemD contract is written and reviewed before implementation expands.
- MemD has a stable scope model and audit trail.
- Retrieval and compaction can be tested independently.
- AgentShell and secrets can integrate with MemD without redefining the memory layer.

## Open Questions

- Whether MemD should live in SQLite first, Supabase first, or a hybrid model.
- Whether a memory item can belong to multiple scopes or must have exactly one primary scope.
- Whether the ranking model is pure heuristic, LLM-assisted, or hybrid.
- Whether MemD should expose user-editable memory facts in the UI on day one.
