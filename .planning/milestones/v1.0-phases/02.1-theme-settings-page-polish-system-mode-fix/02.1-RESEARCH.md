# Phase 02.1: Theme Settings Page Polish + System Mode Fix - Research

**Researched:** 2026-03-19
**Domain:** Tauri v2 theme detection, CSS custom property architecture, Settings UI redesign
**Confidence:** HIGH

## Summary

This phase addresses three interconnected problems discovered during user testing: (1) System mode dark detection fails on Hyprland/Wayland because Tauri's `getCurrentWindow().theme()` reads `gtk-application-prefer-dark-theme`, which is unset on non-GNOME desktops; (2) the Settings > Display page is a flat, unstructured list with repetitive color picker rows; and (3) the "secondary" color picker currently maps to `--accent-blue` (used sparingly in chat bubbles and dashboard) when users expect it to control the far more visible green functional/status color.

The codebase already has a mature theme engine (`theme-store.ts`, `theme-engine.ts`, `theme-definitions.ts`, `themes.ts`) with `useSyncExternalStore` reactivity, per-theme overrides, alpha tint derivation, and ripple transitions. The green color (`--green`, `--green-400`, `--green-500`, `--emerald-*`) is referenced in **45+ component locations** across 25+ files -- it is by far the most visually impactful functional color. The current "secondary" (`--accent-blue`) is used in ~20 locations. The refactoring is straightforward: introduce a `--secondary` variable that replaces hardcoded green references, and rename the current secondary to `--tertiary`.

For system mode, a new Rust Tauri command using `gsettings get org.gnome.desktop.interface color-scheme` (with fallback to `gtk-theme` name inspection) solves the detection problem. The pattern for Tauri commands (`#[tauri::command]` + registration in `generate_handler![]`) is already established in `commands.rs`.

**Primary recommendation:** Add a Rust `detect_system_dark_mode` Tauri command; restructure SettingsDisplay into card-based sections; introduce `--secondary` (default green) and `--tertiary` (default derived-from-accent) color tiers with alpha tint derivation matching the existing accent pattern.

## Architecture Patterns

### Color Variable Architecture (3-Tier)

The new 3-tier system replaces the current ad-hoc green/blue usage:

```
Tier 1: --accent (primary)        Already works. Purple default.
         --accent-dim, --accent-bright, --accent-a10..a40, --purple-a08..a90

Tier 2: --secondary (functional)   NEW. Replaces hardcoded --green.
         --secondary-dim, --secondary-bright
         --secondary-a06, --secondary-a08, --secondary-a12, --secondary-a15,
         --secondary-a20, --secondary-a25, --secondary-a30, --secondary-a45
         Default: #34d399 (current --green value)

Tier 3: --tertiary (accent-blue)   RENAME of current --accent-secondary.
         --tertiary-dim, --tertiary-bright
         Default: derived from --accent (lighten/shift)
```

### Pattern: Alpha Tint Derivation

The theme engine already derives accent alpha tints in `deriveAlphaTints()`. The same pattern must be extended for secondary and tertiary colors. Each color needs:

```typescript
// In theme-engine.ts — new function
export function deriveSecondaryTints(hex: string): void {
  const r = parseInt(hex.slice(1, 3), 16)
  const g = parseInt(hex.slice(3, 5), 16)
  const b = parseInt(hex.slice(5, 7), 16)
  const el = document.documentElement

  const TINTS: [string, number][] = [
    ['secondary-a06', 0.06],
    ['secondary-a08', 0.08],
    ['secondary-a12', 0.12],
    ['secondary-a15', 0.15],
    ['secondary-a20', 0.2],
    ['secondary-a25', 0.25],
    ['secondary-a30', 0.3],
    ['secondary-a45', 0.45],
  ]

  for (const [name, alpha] of TINTS) {
    el.style.setProperty(`--${name}`, `rgba(${r}, ${g}, ${b}, ${alpha})`)
  }
}
```

### Pattern: Tauri Command Registration

Existing pattern from `commands.rs` and `main.rs`:

```rust
// In commands.rs
#[tauri::command]
pub fn detect_system_dark_mode() -> bool {
    #[cfg(target_os = "linux")]
    {
        // 1. Check color-scheme (modern GNOME/freedesktop standard)
        if let Ok(output) = std::process::Command::new("gsettings")
            .args(["get", "org.gnome.desktop.interface", "color-scheme"])
            .output()
        {
            let stdout = String::from_utf8_lossy(&output.stdout);
            if stdout.contains("prefer-dark") {
                return true;
            }
        }

        // 2. Fallback: check gtk-theme name for "dark" substring
        if let Ok(output) = std::process::Command::new("gsettings")
            .args(["get", "org.gnome.desktop.interface", "gtk-theme"])
            .output()
        {
            let stdout = String::from_utf8_lossy(&output.stdout).to_lowercase();
            if stdout.contains("dark") {
                return true;
            }
        }

        false
    }

    #[cfg(not(target_os = "linux"))]
    {
        // On macOS/Windows, Tauri's native detection works fine
        false
    }
}

// In main.rs — add to invoke_handler
.invoke_handler(tauri::generate_handler![
    // ... existing commands ...
    commands::detect_system_dark_mode,
])
```

### Anti-Patterns to Avoid
- **Hardcoded green hex in JSX:** Every instance of `'var(--green)'`, `'var(--green-400)'`, `'var(--emerald-a12)'` etc. must migrate to `--secondary` equivalents. Do NOT leave any hardcoded green references for functional/status usage.
- **Duplicate alpha tint definitions:** Do NOT define secondary alpha tints in globals.css with hardcoded RGB values. Generate them dynamically like accent tints, so they update when the user changes the secondary color.
- **Breaking the ThemeDefinition type:** The `colors` record in each built-in theme already has `'green'` entries. These should be renamed/aliased to `'secondary'` in the definitions, but a migration must handle existing localStorage state.

## Common Pitfalls

### Pitfall 1: Green Alpha Tint Explosion
**What goes wrong:** The codebase has ~15 different green-related alpha tint variables (`--green-400-a06`, `--green-400-a08`, `--green-400-a12`, `--green-400-a14`, `--green-400-a15`, `--green-400-a30`, `--green-400-a45`, `--emerald-a12`, `--emerald-a15`, `--emerald-a20`, `--green-a12`, `--green-a15`, `--green-a25`). All must map to secondary tint equivalents.
**Why it happens:** The green tints evolved organically across different components with slightly different RGB bases (green-400 = `74, 222, 128`, emerald = `59, 165, 92`, green = `52, 211, 153`).
**How to avoid:** Consolidate all green alpha tints to a single set derived from `--secondary`. Use `deriveSecondaryTints()` with a standardized set of alpha levels (06, 08, 12, 15, 20, 25, 30, 45). Update all component references to use the new `--secondary-a*` names.
**Warning signs:** If you see both `var(--green-400-a12)` and `var(--secondary-a12)` in the same file after migration.

### Pitfall 2: gsettings Not Installed
**What goes wrong:** `gsettings` may not be available on minimal Linux installs or non-GNOME desktops without glib2.
**Why it happens:** Arch/CachyOS with Hyprland may not have the full GNOME settings daemon.
**How to avoid:** Wrap the `Command::new("gsettings")` call in error handling. If gsettings fails, fall through to the existing matchMedia fallback. The command should never crash the app.
**Warning signs:** `std::process::Command` returning `Err` or non-zero exit code.

### Pitfall 3: Theme State Migration
**What goes wrong:** Existing users have `overrides.secondary` stored in localStorage (from Phase 02) that maps to the OLD secondary (--accent-blue). After this change, `secondary` means green.
**Why it happens:** The semantic meaning of "secondary" changes.
**How to avoid:** Add a localStorage migration (v6) that renames `overrides.*.secondary` to `overrides.*.tertiary` for all existing theme overrides, and leaves `overrides.*.secondary` unset (so it defaults to green).
**Warning signs:** Users who customized the old secondary color suddenly seeing their chat bubbles turn green.

### Pitfall 4: Light Theme Green Variants
**What goes wrong:** `globals.css` defines different green values for `[data-theme="light"]` (darker shades for contrast: `#059669` vs `#34d399`). If `deriveSecondaryTints()` uses only the base hex, light-mode variants won't have proper contrast.
**Why it happens:** Dark theme greens are bright/pastel; light theme greens need to be darker for WCAG contrast on white.
**How to avoid:** The theme definitions already store per-theme green values (e.g., Default Light has `green: '#059669'`). The `applySecondaryColor()` function should use the resolved theme's green as the default, not a hardcoded hex.
**Warning signs:** Faint, illegible secondary colors on light themes.

### Pitfall 5: Badge Classes with Hardcoded RGB
**What goes wrong:** `.badge-green` in globals.css uses hardcoded `rgba(52, 211, 153, 0.12)` instead of CSS variables.
**Why it happens:** Badge classes predate the theme engine.
**How to avoid:** Rewrite badge-green (and badge-blue) to use `var(--secondary-a12)` and `var(--secondary-bright)` / `var(--tertiary-a12)` etc.
**Warning signs:** Badge colors not updating when user changes secondary/tertiary.

### Pitfall 6: WebKitGTK startViewTransition Crash
**What goes wrong:** The ripple animation crashes on WebKitGTK (Tauri/Linux). Already handled via `isWebKitGTK` check and overlay fallback.
**Why it matters:** Any new theme-switching animations must also respect this guard.
**How to avoid:** Use the existing `performRippleTransition()` function; don't add new View Transitions API calls.

This needs to be augmented with a fallback Tauri command call when the native detection returns 'light' (which may be wrong on Linux):

```typescript
// Enhanced detection for Linux
if (window.__TAURI_INTERNALS__) {
  import('@tauri-apps/api/window').then(({ getCurrentWindow }) => {
    getCurrentWindow().theme().then(async theme => {
      let isDark = theme === 'dark'
      // On Linux, Tauri's native detection often fails — use Rust gsettings fallback
      if (!isDark && navigator.userAgent.includes('Linux')) {
        const { invoke } = await import('@tauri-apps/api/core')
        isDark = await invoke<boolean>('detect_system_dark_mode')
      }
      setOsDarkPreference(isDark)
      if (getThemeState().mode === 'system') applyThemeFromState()
    })
    // Keep the onThemeChanged listener for macOS/Windows
    getCurrentWindow().onThemeChanged(({ payload }) => {
      setOsDarkPreference(payload === 'dark')
      if (getThemeState().mode === 'system') applyThemeFromState()
    })
  })
}
```

export function applySecondaryColor(color: string): void {
  const el = document.documentElement
  el.style.setProperty('--accent-secondary', color)
  el.style.setProperty('--accent-secondary-dim', darken(color, 25))

}

## Open Questions

1. **gsettings on Hyprland without GNOME**
   - What we know: CachyOS with Hyprland likely has `gsettings` via glib2 (WebKitGTK depends on it). The user's system DOES have a dark GTK theme set.
   - What's unclear: Whether `org.gnome.desktop.interface color-scheme` is set vs just `gtk-theme`. Need to check both.
   - Recommendation: Check `color-scheme` first (modern standard), fall back to `gtk-theme` name for "dark" substring. If both fail, return false and let matchMedia try.

2. **Should --green remain as a semantic alias?**
   - What we know: Some uses of green are truly semantic ("success", "online") vs the functional secondary role.
   - What's unclear: Should `var(--green)` remain as an alias for `var(--secondary)`, or should it stay as a fixed "green" for cases where the meaning IS green (success/online)?
   - Recommendation: Make `--green` an alias for `--secondary` everywhere. When the user picks red as their secondary, "online" dots turn red -- this is intentional customization. The user WANTS maximal control. If they pick a weird color, that's their choice.

3. **Border radius / opacity / spacing sliders**
   - What we know: User wants these in an "Advanced" card section. CSS variables `--radius-sm`, `--radius-md`, `--radius-lg`, `--radius-xl` exist. No opacity or spacing override variables exist yet.
   - What's unclear: Exact CSS variables to expose for opacity (glass panel opacity?) and spacing (base spacing multiplier?).
   - Recommendation: Expose `--radius-md` as "Card Radius" (0-24px), `--glass-opacity` as "Panel Opacity" (0.4-1.0 controlling glass-bg alpha), `--spacing-scale` as a multiplier (0.8-1.2) on the spacing variables. Start with radius and opacity; spacing is more complex.
