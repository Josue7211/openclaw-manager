1. Create `frontend/src/lib/theme-scheduling.ts`:

   **approximateSunTimes(date: Date): { sunrise: Date; sunset: Date }**
   - Use the timezone-offset approximation from RESEARCH.md code example
   - Assume ~40 degrees latitude (covers most of US, Europe, East Asia)
   - Calculate solar declination from day of year
   - Calculate hour angle at sunrise/sunset
   - Derive sunrise/sunset hours from solar noon (12:00 local)
   - Return Date objects with hour/minute set

   **checkSchedule(schedule: ThemeSchedule | undefined, now?: Date): string | null**
   - If schedule is undefined or type is 'none', return null
   - If type is 'sunrise-sunset':
     - Call approximateSunTimes(now || new Date())
     - If now is between sunrise and sunset, return schedule.sunriseSunset.dayThemeId
     - Else return schedule.sunriseSunset.nightThemeId
   - If type is 'manual':
     - For each range in schedule.manual:
       - Convert startHour:startMinute and endHour:endMinute to minutes-since-midnight
       - Convert now to minutes-since-midnight
       - Handle wrap-around (e.g. 22:00 to 06:00)
       - If now falls within range, return range.themeId
     - Return null if no range matches

   **startScheduleTimer(): () => void**
   - Set up a 60-second interval that:
     - Calls `getThemeState()` to get current schedule
     - Calls `checkSchedule(state.schedule)`
     - If result is non-null and different from `state.activeThemeId`, call `setActiveTheme(result)` (no click event -- system switch uses center ripple)
   - Return a cleanup function that clears the interval

   **stopScheduleTimer()**
   - Clear the interval (if stored in module-level variable)

   - Test approximateSunTimes for solstices and equinox
   - Test checkSchedule for all three schedule types
   - Do NOT test startScheduleTimer (timer behavior -- tested via integration)
