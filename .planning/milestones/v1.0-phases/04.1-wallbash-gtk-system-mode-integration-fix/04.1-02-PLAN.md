Rewrite the wallbash-related event handling in main.tsx. The current code has 3 problems:
(A) The wallbash-theme-update handler calls setWallbashColors + setGtkThemeMapping + setWallbashColorScheme + setOsDarkPreference as 4 separate calls, allowing resolveThemeDefinition to read partial state if debouncedApply fires between calls.
(B) The debouncedApply has 100ms debounce which is too fast -- it may fire before all state setters complete.
(C) The auto-enable logic `getThemeState().useGtkTheme === undefined` only fires on startup, not when wallbash first becomes active mid-session.

**Specific changes:**

1. **Update imports** -- add `setWallbashState` to the import from theme-engine.ts:
```typescript
import { setOsDarkPreference, setGtkThemeMapping, setWallbashColors, setWallbashColorScheme, setWallbashState, isWallbashActive, wallbashUpdatedRecently } from './lib/theme-engine'
```

2. **Move debouncedApply BEFORE the Tauri IIFE** so it can be used by both Tauri and browser paths. Change the debounce from 100ms to 150ms. This gives the Rust backend's combined event time to arrive and be processed atomically.

```typescript
// Debounced theme apply -- coalesces rapid wallbash/gsettings events.
let _applyTimer: ReturnType<typeof setTimeout> | null = null
const debouncedApply = () => {
  if (_applyTimer) clearTimeout(_applyTimer)
  _applyTimer = setTimeout(() => {
    _applyTimer = null
    applyThemeFromState()
  }, 150)
}
```

3. **Rewrite the wallbash-theme-update event handler** to use setWallbashState:
```typescript
listen<{ colors: WallbashColors; theme: { gtk_theme: string; icon_theme: string; color_scheme: string } }>('wallbash-theme-update', (event) => {
  if (getThemeState().mode !== 'system') return
  const { colors, theme } = event.payload

  // Atomic state update -- all wallbash state changes in one call
  setWallbashState({
    colors: (colors && Object.keys(colors).length > 0) ? colors : undefined,
    colorScheme: theme.color_scheme ? (theme.color_scheme === 'prefer-dark' ? 'prefer-dark' : 'prefer-light') : undefined,
    gtkThemeName: theme.gtk_theme || undefined,
  })

  // Auto-enable GTK theme on first wallbash detection
  const state = getThemeState()
  if (state.useGtkTheme !== true) {
    setUseGtkTheme(true)
  }

  debouncedApply()
})
```

Key change: `state.useGtkTheme !== true` instead of `=== undefined`. This means if the user explicitly set it to false, wallbash events will re-enable it. This is intentional -- if wallbash is actively sending events, the user's desktop IS using wallbash, so the toggle should be on. Users who explicitly disable it would see it re-enable on the next desktop theme switch, which is the correct behavior (they can use dark/light mode instead of system mode if they don't want GTK theme matching).

Wait -- actually that would be annoying. Let me reconsider. If the user explicitly toggles useGtkTheme OFF, we should respect that. The auto-enable should only happen when `useGtkTheme === undefined` (never been set). Change to:

```typescript
if (state.useGtkTheme === undefined) {
  setUseGtkTheme(true)
}
```

This is the same as before but now it fires on EVERY wallbash event (not just startup), so if the user clears their localStorage, it will auto-enable again.

4. **Simplify the gsettings-color-scheme-changed handler** -- this handler should ONLY update osDarkPreference, not touch wallbash state at all. The wallbash file watcher's combined event is the authoritative source for wallbash state.

```typescript
listen<string>('gsettings-color-scheme-changed', (event) => {
  if (getThemeState().mode !== 'system') return
  // Skip if wallbash just updated -- the file watcher event already set osDarkPreference
  if (wallbashUpdatedRecently()) return
  const isDark = event.payload === 'prefer-dark'
  setOsDarkPreference(isDark)
  debouncedApply()
})
```

This is functionally the same as before but now wallbashUpdatedRecently uses 500ms (from plan 01) instead of 5000ms, so gsettings events are only suppressed briefly after a wallbash file write.

5. **Simplify the startup detection IIFE** -- the current startup code in main.tsx (lines 123-145) does too many separate calls. Consolidate using setWallbashState:

```typescript
if (navigator.userAgent.includes('Linux')) {
  (async () => {
    try {
      const { invoke } = await import('@tauri-apps/api/core')
      const colors = await invoke<WallbashColors>('read_wallbash_colors')
      const themeConf = await invoke<{ gtk_theme: string; color_scheme: string }>('read_theme_conf')

      const hasWallbash = colors && Object.keys(colors).length > 0

      // Atomic initial state setup
      setWallbashState({
        colors: hasWallbash ? colors : undefined,
        colorScheme: themeConf?.color_scheme ? (themeConf.color_scheme === 'prefer-dark' ? 'prefer-dark' : 'prefer-light') : undefined,
        gtkThemeName: themeConf?.gtk_theme || undefined,
      })

      // Also set osDarkPreference from theme.conf color_scheme
      if (themeConf?.color_scheme) {
        setOsDarkPreference(themeConf.color_scheme === 'prefer-dark')
      }

      // Auto-enable GTK theme mode on first detection
      if ((hasWallbash || themeConf?.gtk_theme) && getThemeState().useGtkTheme === undefined) {
        setUseGtkTheme(true)
      }

      if (getThemeState().mode === 'system') applyThemeFromState()
    } catch { /* wallbash files not present */ }
  })()
}
```

6. **Keep the onThemeChanged handler as-is** -- it correctly uses `wallbashUpdatedRecently()` to skip when wallbash just updated (now with the shorter 500ms window from plan 01).

7. **Keep the 3s polling interval for GTK theme name changes** -- it already has the `if (isWallbashActive()) return` guard. This poll only catches non-wallbash GTK theme name changes (e.g., user switches from Adwaita to Dracula via gsettings).
