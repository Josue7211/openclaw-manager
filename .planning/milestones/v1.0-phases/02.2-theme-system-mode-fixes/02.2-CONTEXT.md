# Phase 02.2: Theme System Mode Fixes - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning
**Source:** User testing feedback from Phase 02.1

<domain>
## Phase Boundary

Fix System mode to truly mirror the desktop theme: show only the active GTK theme card, read live Wallbash colors dynamically, sync instantly with desktop transitions, auto-switch presets on mode change, and ensure all light themes have readable text. Also handle Windows/macOS gracefully where GTK detection isn't available.

This phase does NOT include: live preview pane (future), GTK parity features (border colors, blur, gaps — future), or new customization features from brainstorm.

</domain>

<decisions>
## Implementation Decisions

### System Mode Preset Display (SYSMODE-01)
- When System mode is active, show ONLY the active system theme card (e.g., "Material Sakura") with its color swatches
- All other preset cards disappear — system controls the theme, user can't manually pick
- User can still customize accent/secondary/tertiary colors on top of the system theme
- Label the card clearly: "System Theme: Material Sakura" or similar

### Dark↔Light Auto-Switch (SYSMODE-02)
- When user switches mode (Dark→Light or Light→Dark), auto-switch to counterpart preset
- Counterpart mapping: gruvbox-dark↔gruvbox-light, catppuccin-mocha↔catppuccin-latte, solarized-dark↔solarized-light, default-dark↔default-light, high-contrast-dark↔high-contrast-light
- Themes without counterpart (dracula, nord, rose-pine, tokyo-night, etc.) fall back to default-dark/default-light
- The auto-switch should feel seamless — part of the mode toggle animation

### Live Wallbash Colors (SYSMODE-03 + SYSMODE-04)
- When GTK theme is "Wallbash-Gtk", DO NOT use a static preset
- Instead, read live colors from `~/.config/hypr/themes/colors.conf` which Wallbash regenerates on every change
- Parse the wallbash color groups (pry1/txt1/1xa1-1xa9 etc.) and map to app CSS variables
- When Wallbash switches between dark/light/auto mode, the gtk color-scheme changes — the app should re-read GTK colors and update CSS variables WITHOUT switching the app's Dark/Light mode toggle
- The app just mirrors whatever colors the desktop is using

### Instant Sync via File Watcher (SYSMODE-05)
- Replace 1-second gsettings polling with file watching on `~/.config/hypr/themes/colors.conf`
- Use Tauri fs plugin's `watch` or native `notify` crate to detect file changes
- When the file changes, immediately re-read colors and apply — sync with desktop ripple transition
- Fall back to 1s polling if file watching isn't available

### Light Mode Contrast Fix (SYSMODE-06)
- All light theme presets must pass WCAG AA contrast:
  - text-primary against bg-base: ≥ 7:1 (AAA)
  - text-secondary against bg-base: ≥ 4.5:1 (AA)
  - text-muted against bg-base: ≥ 3:1 (AA large)
- Light theme text-primary should be ≤ #333333, text-secondary ≤ #555555
- Fix: Default Light, Material Sakura, Catppuccin Latte, Solarized Light, Gruvbox Light, High Contrast Light
- Existing automated WCAG tests (120 tests from contrast audit) should catch regressions

### Windows/macOS Fallback (SYSMODE-07)
- On Windows/macOS where GTK detection isn't available, System mode falls back to OS dark/light detection
- Show dark presets in dark system mode, light presets in light system mode (same filtering as Dark/Light manual modes)
- No single "System Theme" card on these platforms — just filtered presets

### Claude's Discretion
- Exact wallbash color group → CSS variable mapping
- File watcher implementation details (notify crate vs Tauri fs plugin)
- "System Theme" card visual design
- Transition animation timing to sync with desktop

</decisions>

<code_context>
## Existing Code Insights

### Current GTK Detection Flow
1. `main.tsx` calls `detect_gtk_theme` (Tauri command) on startup
2. `setGtkThemeMapping()` maps GTK name → built-in preset ID via regex
3. `resolveThemeDefinition()` uses mapped preset when mode=system
4. 1s polling re-checks GTK theme + color-scheme

### What Needs to Change
- `resolveThemeDefinition()` needs a Wallbash-specific path that reads live colors instead of mapping to a static preset
- `ThemePicker.tsx` and `SettingsDisplay.tsx` need system-mode-aware rendering (show only active theme card)
- `main.tsx` polling should be replaced with file watcher for instant sync
- New Tauri command needed: `read_wallbash_colors` that parses colors.conf
- Light theme definitions need text color adjustments

</code_context>
