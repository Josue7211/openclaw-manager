<objective>
Fix 3 Bjorn backend endpoints that return minimal JSON instead of full module objects, causing frontend runtime crashes on update/toggle/rollback operations.

Purpose: The frontend bjorn-store.ts calls `registerBjornModule(result.module)` after every mutation, but update_module returns `{ ok, version }`, toggle_module returns `{ ok, enabled }`, and rollback_module returns `{ ok, version }` -- all missing the `module` key. This makes BJORN-10 (persistence), BJORN-11 (toggle), and BJORN-12 (rollback) only partially functional.

Output: All three endpoints return `{ "module": { ...full BjornModule fields } }` matching the create_module pattern. Additionally, the `defaultSize` deserialization is fixed to accept the nested `{ w, h }` object the frontend sends.
</objective>

**4. Fix `defaultSize` deserialization in `CreateModuleBody` and `UpdateModuleBody`:**

The frontend sends `defaultSize: { w: 3, h: 3 }` as a nested object, but the Rust structs use flat `defaultSizeW`/`defaultSizeH` fields which never match. Add a `defaultSize` field that accepts the nested object and falls back to the flat fields.

Add a helper struct and flatten pattern:
```rust
#[derive(Debug, Deserialize)]
struct SizeObj {
    w: Option<i64>,
    h: Option<i64>,
}
```

Then in `CreateModuleBody`, add:
```rust
#[serde(rename = "defaultSize")]
default_size: Option<SizeObj>,
```

And update the extraction logic to prefer `default_size.w` over `default_size_w`:
```rust
let w = body.default_size.as_ref().and_then(|s| s.w)
    .or(body.default_size_w)
    .unwrap_or(3);
let h = body.default_size.as_ref().and_then(|s| s.h)
    .or(body.default_size_h)
    .unwrap_or(3);
```

Apply the same pattern to `UpdateModuleBody`.

```rust
fn module_row_to_json(

        "updatedAt": row.12,
    })
}
```

Then use `module_row_to_json(&row)` in all 4 places. For `list_modules`, refactor the `.map()` closure to call this helper. For the 3 mutation endpoints, use `fetch_one` + helper.

Also update `create_module` to use the helper for consistency (construct tuple from local variables and call helper, OR just keep it inline since it builds from body fields not a DB row -- use your judgment, consistency is the priority).
